## Use When

- Building golden **tasks** (not prompts) — each one a complete agent scenario from goal to final state.
- Measuring agent-specific metrics: task success rate, intervention rate, irreversible-action rate, step efficiency, tool-choice quality.
- Standing up **replay-based eval**: re-run a recorded trace against a new prompt / model / tool version and diff.
- Wiring agent eval into the CI gate so prompt or tool changes that regress agent performance block merge.
- Per-tenant agent eval before flipping a flagship customer to a new agent version.

## Do Not Use When

- The task is single-shot prompt eval — `ai-eval-harness`.
- The task is the eval *concept* — `ai-evaluation`.
- The task is hallucination measurement specifically — `ai-hallucination-slo-and-grounding`.
- The task is observability of running traffic — `ai-agent-observability-and-replay`.

## Required Inputs

- Agent runtime with persisted task traces (`ai-agent-runtime-architecture`, `ai-agent-observability-and-replay`).
- Tool registry (`ai-agent-tool-catalogue-and-action-gating`).
- Baseline eval harness (`ai-eval-harness`).
- Recorded production traces (sanitised) to seed goldens.

## Workflow

1. Read this `SKILL.md`.
2. Build **golden tasks** (§1). See `references/golden-tasks-construction.md`.
3. Define **agent-specific metrics** (§2): task success, step efficiency, tool-choice quality, irreversible-action rate, intervention rate, time-to-complete.
4. Implement **trajectory scoring** (§3) — judge-LLM or rule-based per metric.
5. Implement **replay-based eval** (§4). See `references/replay-based-eval.md`.
6. Wire **CI gate** (§5) — block merge on agent eval regression.
7. Implement **per-tenant gate** (§6) — flagship sign-off before agent promotion.
8. Apply anti-patterns (§7).

## Quality Standards

- Each agent feature has a golden set of ≥ 30 task scenarios covering happy path, edge cases, adversarial inputs, tool failures.
- Each flagship tenant has a per-tenant golden subset (≥ 10 scenarios) derived from their support tickets and use cases.
- Metric definitions are written down and stable; dashboards never break on metric renames.
- CI gate blocks merge if any metric regresses below threshold.
- Replay eval is **deterministic within tolerance** — re-running the same trace + same prompt + same tools (mocked) gives consistent metric scores.
- Judge-LLM agreement with humans on agent trajectories ≥ 75% (lower bar than single-shot because trajectories are longer); recalibrate if below.
- Irreversible-action rate on the golden suite is **exactly 0** for actions outside the expected scope.

## Anti-Patterns

- "Did the final answer look right?" as the only metric. Misses step efficiency, tool misuse, and unnecessary irreversibles.
- Goldens that score only success/failure with no step trajectory. Cannot diagnose why an agent regressed.
- Replay eval that calls live tools. Distorts cost, breaks idempotency, may charge real cards.
- Eval suite runs only against the flagship model. Cheaper models look broken because nobody tests them.
- CI gate that *warns* on regression but doesn't block. No accountability.
- Goldens generated by the agent under test. Tautology.
- Per-tenant goldens copied without sanitisation. PII leak into eval dataset.

## §1 Golden Tasks (not Prompts)

A golden task is a **scenario**, not a prompt:

```yaml
feature: support_copilot
scenario_id: refund_dispute_001
goal: "Customer claims double charge; verify and refund if valid."
preconditions:
  customer_id: 88
  invoice_ids: [1234, 1235]
  expected_kb_articles: [refund_policy]
fixtures:
  - tool: charge_lookup
    request: {invoice_id: 1234}
    response: {amount: 89.00, status: paid, charged_at: "2026-05-10T10:00:00Z"}
  - tool: charge_lookup
    request: {invoice_id: 1235}
    response: {amount: 89.00, status: paid, charged_at: "2026-05-10T10:02:00Z", note: "second charge"}
expected_outcome:
  state: COMPLETED
  irreversible_actions_taken: ["payment_refund"]
  irreversible_actions_args:
    - tool: payment_refund
      args: {invoice_id: 1235, amount: 89.00}
  steps_used_range: [4, 8]
  intervention_required: false
  hallucinated_tool_args_max: 0
  customer_facing_response_must_contain:
    - "double charge"
    - "refund"
  customer_facing_response_must_not_contain:
    - "<unknown>"
    - "I don't have access"
```

Full construction guide in `references/golden-tasks-construction.md`.

## §2 Agent-Specific Metrics

| Metric | Definition | Target |
|---|---|---|
| **Task success rate** | % of tasks reaching `COMPLETED` with expected outcome | ≥ 80% (feature-dependent) |
| **Step efficiency** | (steps used) / (optimal step count). Optimal from golden. | ≤ 1.5× |
| **Tool-choice quality** | % of steps where the agent picked an allowed, sensible tool | ≥ 90% |
| **Hallucinated tool args rate** | % of tool calls where args don't validate against schema or reference non-existent IDs | ≤ 1% |
| **Irreversible-action rate (off-script)** | Count of irreversible actions taken that the golden didn't expect | exactly 0 |
| **Intervention rate** | % of tasks that required HITL approval | feature-tuned (low for analyst agents, high for legal agents) |
| **Time-to-complete** | Median wallclock seconds | feature-tuned |
| **Cost per successful task** | USD / completed task | feature-tuned |
| **Approval edit rate** | % of approvals where the user edited args before approving | ≤ 20% (high = agent generates wrong args) |

## §3 Trajectory Scoring

Scoring is per-trajectory, not just per-final-output:

```python
def score_trajectory(actual_trace: Trace, golden: GoldenScenario) -> Scores:
    scores = Scores()
    scores.completed = actual_trace.terminal_state == golden.expected_outcome.state
    scores.steps_in_range = (
        golden.expected_outcome.steps_used_range[0]
        <= actual_trace.steps_used
        <= golden.expected_outcome.steps_used_range[1]
    )
    scores.step_efficiency = actual_trace.steps_used / golden.optimal_steps
    # Tool-choice quality: each step's tool against allowed list
    allowed = set(golden.expected_outcome.allowed_tools or tool_registry.list_allowed_for(golden))
    scores.tool_choice_quality = sum(
        1 for s in actual_trace.steps if s.tool_name in allowed
    ) / max(1, len(actual_trace.steps))
    # Hallucinated args: validate args against schema + entity existence
    scores.hallucinated_args = sum(
        1 for s in actual_trace.steps if has_invalid_or_phantom_args(s)
    )
    # Off-script irreversibles
    expected_irr = canonical_set(golden.expected_outcome.irreversible_actions_taken)
    actual_irr = canonical_set(s.tool_name for s in actual_trace.steps
                                if registry.get(s.tool_name).reversibility == 'irreversible')
    scores.off_script_irreversibles = list(actual_irr - expected_irr)
    # Response content (judge-LLM)
    scores.response_quality = judge_response(actual_trace.final_response, golden)
    return scores
```

Run on every golden scenario; aggregate per feature, per model.

## §4 Replay-Based Eval

Take a recorded production trace and "replay" it against a new agent config. The tools are mocked from the trace; the LLM is the new candidate; the observed differences are the regression / improvement.

Use cases:
- Test a new prompt against last week's traffic.
- Test a model upgrade.
- Test a tool-registry change.
- "What would the agent have done differently?" debugging.

Full implementation in `references/replay-based-eval.md`.

## §5 CI Gate

```yaml
name: agent-eval
on: [pull_request]
jobs:
  agent-eval:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Run agent eval
        run: python -m eval.agent_eval --features=$(cat .github/changed_agent_features.txt) --report=agent-eval.json
      - name: Check thresholds
        run: python -m eval.check_thresholds agent-eval.json --thresholds=eval/thresholds.yaml
```

`eval/thresholds.yaml`:

```yaml
default:
  task_success_rate_min: 0.80
  tool_choice_quality_min: 0.90
  hallucinated_args_max: 0.01
  off_script_irreversibles_max: 0
  step_efficiency_max: 1.5

per_feature:
  support_copilot:
    task_success_rate_min: 0.85
  log_investigator:
    task_success_rate_min: 0.70
    step_efficiency_max: 2.0
```

## §6 Per-Tenant Gate

Before promoting a new agent version to a flagship tenant:

```bash
python -m eval.agent_eval \
  --features=support_copilot \
  --tenant_goldens=goldens/per-tenant/acme.yaml \
  --candidate=v2.3
```

Sign-off requires:
- All tenant-specific scenarios pass.
- Aggregated metrics within their tenant's negotiated SLA.
- A signed report attached to the rollout PR.

## §7 Anti-Patterns

- Single-number "success rate" with no trajectory scoring. Misses why.
- Replay using live tools. Cost + side effects + flake.
- Eval only against the flagship model.
- CI warns only. No accountability; regressions ship.
- Goldens generated by the agent under test.
- Per-tenant gate skipped because "it's just a small change".
- Metrics on dashboards but no thresholds. Trends drift without triggering action.
