## Use When

- Standing up the eval harness as a control-plane service before the first enterprise customer's procurement reviews quality.
- Adding a CI gate that blocks deploys regressing the golden suite.
- Building per-tenant goldens for flagship customers (their support tickets, their docs, their tone).
- Wiring a judge-LLM pipeline to score open-ended outputs at scale.
- Detecting drift between staging and production output distributions on the same prompts.

## Do Not Use When

- The task is conceptual evaluation strategy — `ai-evaluation`.
- The task is the prompt design itself — `ai-prompt-engineering`.
- The task is hallucination measurement specifically — `ai-hallucination-slo-and-grounding`.

## Required Inputs

- Inventory of AI features and their target metrics (correctness, faithfulness, tone, format compliance).
- Sample of production traffic (sanitised) to seed goldens.
- Per-tenant golden requirements from flagship customers.
- Prompt registry from `ai-on-saas-architecture`.

## Workflow

1. Read this `SKILL.md`.
2. Build the **golden dataset structure** (§1) — per-feature, per-tenant, versioned.
3. Define **metrics per feature** (§2) — what "good" means.
4. Wire **judge-LLM scoring** (§3) for open-ended outputs.
5. Implement the **CI gate** (§4) — block prompt/model changes that regress.
6. Wire **production sampling + drift detection** (§5).
7. Implement **per-tenant eval gates** (§6) — flagship sign-off.
8. Apply anti-patterns (§7).

## Quality Standards

- Every published prompt has a golden suite of ≥ 50 examples.
- Every flagship tenant has a per-tenant golden subset of ≥ 30 examples.
- CI gate blocks merge on regression of any metric below threshold.
- Judge-LLM agreement with humans ≥ 80% on the calibration set; otherwise judge is recalibrated.
- Drift detector runs daily; alerts on > 2-sigma shift in any metric.
- Per-tenant eval runs before promoting a prompt to that tenant.

## Anti-Patterns

- "Eyeball it in staging." No baseline; no regression detection.
- Goldens generated by the same model that produces production output. Tautology.
- One pooled golden dataset across all tenants. Hides per-tenant regressions.
- Judge-LLM with no human calibration set. Trusts a black box.
- CI gate that warns but doesn't block. No accountability.
- Goldens checked in as JSON blobs with no schema, no versioning, no review.

## Outputs

- Golden dataset directory (per-feature, per-tenant) with a schema.
- Metric definitions + reference scorers.
- Judge-LLM prompt + calibration set.
- CI workflow (eval-on-PR + nightly drift).
- Per-tenant eval runner.

## §1 Golden Dataset Structure

A repo layout:

```
eval/
├── goldens/
│   ├── support-copilot.answer/
│   │   ├── core.jsonl                 # ~200 examples (broad coverage)
│   │   ├── adversarial.jsonl          # known failure modes
│   │   ├── regression.jsonl           # bugs we fixed; must stay fixed
│   │   ├── tenants/
│   │   │   ├── acme.jsonl             # flagship tenant's examples
│   │   │   └── globex.jsonl
│   │   └── manifest.yaml              # version, metrics, judge prompt id
│   └── analytics.sql-from-question/
│       └── ...
├── scorers/                            # deterministic + ML scorers
│   ├── exact_match.py
│   ├── json_schema_valid.py
│   ├── citation_grounded.py
│   └── judge_llm.py
└── runner/
    ├── run.py                          # CLI: eval run --feature X
    └── ci.py                           # CI entry point
```

Golden row schema (JSONL):
```json
{
  "id": "sup_ans_0001",
  "input": {"user_question": "..."},
  "retrieval_fixture": {"chunks": [...]},          // for RAG eval, frozen
  "expected": {
    "must_include": ["refund policy", "30 days"],
    "must_not_include": ["legal advice"],
    "answer_distribution": "concise|moderate",
    "format": "markdown",
    "citation_required": true
  },
  "metadata": {"persona": "free-user", "difficulty": "medium", "added": "2026-03-12", "added_by": "pb"}
}
```

Versioned via git. Adding rows is a PR; removing rows requires a justification.

## §2 Metrics per Feature

Each feature picks a small set of metrics. Default starter set:

| Metric | Scorer | Threshold |
|---|---|---|
| Correctness | judge_llm with reference | ≥ 0.85 mean |
| Faithfulness (no hallucination) | citation_grounded + judge | ≥ 0.95 mean |
| Format compliance | json_schema_valid / regex | 100% |
| Required-content present | exact_match phrase list | ≥ 0.98 |
| Forbidden-content absent | regex | 100% |
| Latency p95 (model side) | timing | feature-specific |
| Cost per generation (median) | from gateway | < target USD |

Don't add a metric you can't action.

## §3 Judge-LLM Patterns

See `references/llm-as-judge-patterns.md`. Summary:

- Use a different (and ideally stronger) model than the candidate.
- Provide rubric, reference answer (if any), and 1-shot calibration example.
- Score on a small ordinal scale (1–4); avoid 1–10.
- Calibrate against ≥ 100 human-scored examples; reject the judge if agreement < 0.8 (Cohen's kappa).
- Sample-rate: 100% on goldens, 1–5% on production for drift.

## §4 CI Gate

`.github/workflows/ai-eval.yml`:

```yaml
name: ai-eval
on:
  pull_request:
    paths:
      - 'prompts/**'
      - 'eval/goldens/**'
      - 'gateway/providers/**'

jobs:
  goldens:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: pip install -r eval/requirements.txt
      - run: python -m eval.runner.ci --features changed --compare-base
      - run: python -m eval.runner.ci --report-pr
```

Behaviour:
- Identify which features are affected by the PR (changed prompt id, changed model adapter).
- For each feature, run goldens on base branch and PR branch.
- Compare metrics; any regression below threshold = fail.
- Post a comment on the PR with per-metric deltas and links to failing examples.

See `references/eval-ci-gate.md` for the full implementation.

## §5 Production Sampling + Drift

The gateway samples N% of production requests (`ai-on-saas-architecture` §5) to an eval queue. Nightly:

1. Run judge-LLM on the sample.
2. Compute metric distributions per (feature, model, prompt_version).
3. Compare with rolling 7-day baseline.
4. Alert on > 2-sigma shifts or threshold breaches.

Dashboard panels: faithfulness over time per feature; correctness per prompt version; drift heatmap by tenant tier.

## §6 Per-Tenant Eval Gate

Before promoting a prompt change to a flagship tenant:

```bash
eval run \
  --feature support-copilot.answer \
  --prompt-version v18 \
  --tenant acme \
  --baseline-prompt-version v17
```

The runner uses the tenant-specific golden set and the tenant's KB partition. Promotion requires:
- No metric regression.
- Tenant-specific tone metric passes.
- Tenant-specific compliance scorer passes (e.g., must not mention banned terms).

Recorded in the prompt registry as `tenant_prompt_pin.eval_passed_at`.

## §7 Anti-Patterns

- Goldens generated by the candidate model. Eval looks great; production is broken.
- One judge model used for years. Drift, contamination.
- Per-tenant goldens that are just product goldens with the tenant name swapped. No real signal.
- No regression suite (`regression.jsonl`). Old bugs return.
- Metrics added for a one-off project, never removed. Dashboard rot.
- CI runs goldens only against a single example. Statistically noisy.
- Drift alerts that page on noise. Tune thresholds; require persistence.

## §8 Agent Eval Section

Single-shot prompt eval is insufficient for agentic features. An agent's "answer" is a **trajectory** — a sequence of plans, tool calls, observations, and a final response. The metrics, goldens, and replay pipeline differ.

The full agent eval skill is `ai-agent-eval`. This section is the bridge.

### Metrics specific to agents

| Metric | Definition |
|---|---|
| Task success rate | % of tasks reaching `COMPLETED` with the expected outcome |
| Step efficiency | (steps used) / (optimal step count) |
| Tool-choice quality | % of steps where the agent picked an allowed, sensible tool |
| Hallucinated tool args rate | % of tool calls with invalid args or phantom IDs |
| Irreversible-action rate (off-script) | Count of irreversibles taken not in the golden's expected set — target exactly 0 |
| Intervention rate | % of tasks requiring HITL approval |
| Time-to-complete | Median wallclock seconds |
| Cost per successful task | USD / completed task |
| Approval edit rate | % of approvals where the user edited args before approving |

### Goldens are scenarios, not (prompt, answer) pairs

An agent golden defines: starting state, fixtures (mocked tool responses), the user goal, the expected trajectory bounds (allowed tools, required irreversibles, forbidden irreversibles, step-count range, intervention required, final-response assertions). Full anatomy in `ai-agent-eval/references/golden-tasks-construction.md`.

### Replay-based eval

For agents, **replay** is a primary eval mechanism: take a recorded production trace, substitute the candidate prompt / model / tool registry, mock the tools with recorded observations, re-run the loop, and diff. Lets you test changes against thousands of real scenarios without authoring goldens. Full pipeline in `ai-agent-eval/references/replay-based-eval.md`.

### CI gate additions

Thresholds (defaults):
- `task_success_rate ≥ 0.80`
- `tool_choice_quality ≥ 0.90`
- `hallucinated_args ≤ 0.01`
- `off_script_irreversibles = 0` (exact)
- `step_efficiency ≤ 1.5`

Replay CI: `same_terminal_rate ≥ 0.92`, `off_script_irreversibles_rate ≤ 0.005`, `unmatched_tool_calls_rate ≤ 0.05`.

### Per-tenant agent gate

Before promoting a new agent version to a flagship tenant, run both the tenant-specific agent goldens (the tenant's representative tasks) and a replay of the tenant's recent production traces. Sign-off attaches to the rollout PR.
