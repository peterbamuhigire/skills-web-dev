# Golden Dataset Construction — Reference

How to build datasets that meaningfully signal regressions and survive months of feature evolution.

## Sources of examples

Use all four; in this order of trustworthiness:

1. **Production traffic, human-labelled.** Sanitised, sampled across personas / difficulty / tenant tiers. Highest signal.
2. **Customer-reported bugs.** Every "the bot got this wrong" ticket becomes a regression row after fix.
3. **Domain experts hand-written.** For coverage gaps and adversarial.
4. **Synthetic** by a different model than the candidate, then human-reviewed. Fills long-tail gaps; cheapest.

Avoid:
- Examples generated by the candidate model with the candidate prompt.
- Examples from public benchmarks unless validated for distribution match.

## Size and coverage

- **Core**: 150–300 examples per feature.
- **Adversarial**: 30–60 known-tricky cases.
- **Regression**: 1 row per bug ever fixed. Grows monotonically.
- **Per-tenant**: 30–80 per flagship.

Coverage axes to balance:
- Persona (admin, end-user, integration).
- Difficulty (easy / medium / hard).
- Input length (short / medium / long).
- Locale / language (if multi-language).
- Topic / intent.
- Edge cases (empty input, very long, off-topic, adversarial).

## Schema (JSONL)

```json
{
  "id": "stable-string-id",
  "input": { "<feature-specific>": "..." },
  "context": { "kb_chunks": [...], "user_persona": "...", "locale": "en-GB" },
  "expected": {
    "answer_text": "...",          // optional reference
    "must_include": ["..."],
    "must_not_include": ["..."],
    "format": "markdown|json|...",
    "json_schema": {...},
    "citation_required": true,
    "tone": "concise"
  },
  "weight": 1.0,
  "metadata": {
    "source": "production|expert|synthetic|bug-fix",
    "added": "2026-03-12",
    "added_by": "pb",
    "tenant": null,
    "difficulty": "medium"
  }
}
```

`weight` lets adversarial rows count for more in aggregate metrics.

## Maintaining

- **Add** rows: a PR with the rows + a justification in the description.
- **Remove**: requires explicit reason (e.g., feature retired, source policy change). Recorded in `eval/goldens/CHANGELOG.md`.
- **Audit yearly**: re-review rows for staleness (e.g., outdated policy citations).
- **Per-tenant maintenance**: with each flagship's quarterly review, refresh 10% of their rows.

## Avoiding poisoning

- Goldens go through PII review before commit.
- If sourced from production, the `request_id` is recorded so audit log can prove provenance + consent.
- Synthetic rows are flagged; never > 30% of any golden set.

## When goldens lie

If production performance is great but goldens regress (or vice versa):

- The golden distribution drifted from production. Refresh.
- The metric is measuring the wrong thing. Adjust scorer.
- Production sample is biased (e.g., one tenant's traffic dominates). Stratify.

## Test the goldens themselves

A meta-test: run the *current* production output against the goldens monthly. If it doesn't pass the threshold, the goldens are mis-thresholded or production is silently broken.
