# Operating Rhythm

Use this as a default cadence for a software team.

## Weekly

- planning or reprioritization
- design and architecture review for risky changes
- release readiness check for pending deployments
- one-to-ones for unblockers, context, and growth

## Per Change

- clear scope and owner
- explicit risk and validation plan
- code review with migration and rollback awareness
- release evidence for high-risk changes

## Per Incident or Release

- verify impact, timeline, and owner
- stabilize first, then diagnose deeply
- capture lessons in tooling, tests, docs, or simplified design

## Per Quarter

- review system bottlenecks
- review team topology and ownership seams
- retire process that no longer serves outcomes
- invest in documentation, tooling, and training where compounding value exists
